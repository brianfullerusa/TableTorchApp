## Phase 1: Critical Bug Fixes (Do Immediately)

### Issue #1 -- Palette switch resets color index to 0
- **Status:** COMPLETE (2026-03-25) -- Approved by 2 Architects + 2 Code Reviewers
- **Severity:** Critical
- **File:** `PreferencesManager.kt:337`
- **Description:** When changing palettes, the selected color position resets to index 0 instead of preserving the user's current position. Line 337 hardcodes `LAST_SELECTED_COLOR_INDEX = 0`. Since all palettes have exactly 6 colors (PALETTE_SIZE=6), any index 0-5 is always valid across palettes.
- **Fix:** Remove line 337. Add `coerceIn(0, PALETTE_SIZE-1)` on the read path in `settingsFlow` for defense-in-depth.
- **Consensus:** All 15 reviewers agree.

### Issue #2 -- Use collectAsStateWithLifecycle()
- **Status:** COMPLETE (2026-03-25) -- Approved by 2 Architects + 2 Code Reviewers
- **Severity:** Critical
- **File:** `NavGraph.kt:41-42`
- **Description:** `collectAsState()` continues collecting when app is backgrounded, wasting battery on a torch app with sensor data. Should use lifecycle-aware version.
- **Fix:** Replace `collectAsState()` with `collectAsStateWithLifecycle()`. Dependency already exists.
- **Consensus:** All 15 reviewers agree.

### Issue #3 -- Version number review
- **Status:** NO CHANGE NEEDED (2026-03-25)
- **Severity:** N/A
- **File:** `build.gradle.kts:31-32`
- **Description:** Reviewers flagged `versionCode=1` / `versionName="1.0"` based on git history referencing "3.0 redesign." However, v3.0 refers to the iOS app design being ported -- this is the first Android release. `versionCode` must be 1 for initial Play Store upload (Google Play requirement). `versionName = "1.0"` is correct as this is Android v1. iOS and Android version numbers are independent.
- **Decision:** Current values are correct. No change required.
